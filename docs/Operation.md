# 공공직군 행정업무 슈퍼앱 Operation

## 1. 문서 목적

본 문서는 공공직군 행정업무 슈퍼앱의 개발 및 운영 실행 방법, 주요 기능 사용 절차, 자주 발생하는 오류와 대응 방법을 정리한다. 현재 저장소는 초기 골격 단계이므로 실행 명령은 확인된 프론트엔드 패키지와 권장 백엔드 구조를 기준으로 작성한다.

## 2. 사전 준비

### 2.1 필수 도구

- Node.js
- npm
- Python
- Python 가상환경 도구
- PostgreSQL
- Redis

### 2.2 권장 환경 변수

백엔드 운영 시 다음 환경 변수를 사용한다.

```env
APP_ENV=local
APP_SECRET_KEY=change-me
DATABASE_URL=postgresql://user:password@localhost:5432/admin_superapp
REDIS_URL=redis://localhost:6379/0
FILE_STORAGE_PATH=./storage
LLM_PROVIDER=openai
LLM_API_KEY=
NEWS_COLLECT_TIME=08:00
```

실제 운영 환경에서는 `.env` 파일을 저장소에 커밋하지 않는다. 비밀키와 API 키는 Secret Manager 또는 운영 환경의 안전한 설정 저장소로 관리한다.

## 3. 로컬 실행 방법

### 3.1 프론트엔드

현재 `frontend/package.json`에는 React, React DOM, Vite, TypeScript 패키지가 포함되어 있다. 개발 서버 스크립트가 없다면 다음 스크립트를 추가한다.

```json
{
  "scripts": {
    "dev": "vite",
    "build": "vite build",
    "preview": "vite preview"
  }
}
```

실행:

```powershell
cd frontend
npm install
npm run dev
```

기본 접속 주소:

```text
http://localhost:5173
```

### 3.2 백엔드

백엔드는 FastAPI 기반을 권장한다. 엔트리 파일이 준비된 뒤 다음 방식으로 실행한다.

```powershell
cd backend
.\.venv\Scripts\activate
uvicorn app.main:app --reload --host 127.0.0.1 --port 8000
```

기본 API 주소:

```text
http://localhost:8000
```

API 문서:

```text
http://localhost:8000/docs
```

### 3.3 워커와 스케줄러

엑셀 대용량 처리, 뉴스 수집, 매뉴얼 인덱싱은 백그라운드 작업으로 분리한다.

예시:

```powershell
cd backend
.\.venv\Scripts\activate
python -m app.workers.worker
python -m app.workers.scheduler
```

실제 명령은 워커 구현 방식에 맞춰 확정한다.

## 4. 기능 사용법

### 4.1 팀 일정 관리

1. 캘린더 화면으로 이동한다.
2. 월간, 주간, 일간 보기 중 필요한 보기를 선택한다.
3. 일정 등록 버튼을 누른다.
4. 유형, 기간, 제목, 공개 범위, 설명을 입력한다.
5. 저장 후 캘린더에 일정이 반영되었는지 확인한다.

운영 기준:

- 휴가 사유 등 민감한 설명은 비공개 또는 제한 공개로 등록한다.
- 팀장 권한으로 조회 또는 변경한 이력은 반드시 감사 로그에 남긴다.
- 장기 출장, 교육 등 업무 공백을 만드는 일정은 팀 필터에서 쉽게 확인할 수 있어야 한다.

### 4.2 엑셀 업무 자동화

파일 분리:

1. 엑셀 자동화 화면으로 이동한다.
2. 파일을 업로드한다.
3. 미리보기에서 컬럼이 정상 인식되었는지 확인한다.
4. 분리 기준 컬럼을 선택한다.
5. 빈 값 처리와 파일명 규칙을 선택한다.
6. 실행 후 결과 파일을 다운로드한다.

파일 병합:

1. 여러 파일을 업로드한다.
2. 기준 컬럼 구조를 확인한다.
3. 누락 컬럼 처리 방식을 선택한다.
4. 중복 제거 기준을 선택한다.
5. 실행 후 병합 결과와 오류 리포트를 확인한다.

운영 기준:

- 개인정보가 포함된 파일은 필요한 경우에만 업로드한다.
- 처리 완료 후 원본 파일 자동 삭제 정책을 적용한다.
- 오류 리포트는 결과 파일과 함께 보관하되 보관 기간을 제한한다.

### 4.3 민원 대응 챗봇

1. 관리자가 민원 매뉴얼을 업로드한다.
2. 시스템이 문서를 파싱하고 검색 인덱스를 생성한다.
3. 사용자가 민원 내용을 입력한다.
4. 응대 유형을 선택한다.
5. 답변 초안, 근거 문서, 검토 필요 여부를 확인한다.
6. 담당자가 내용을 검토한 뒤 실제 응대에 사용한다.

운영 기준:

- 챗봇 답변은 최종 답변이 아니라 초안이다.
- 근거 문서가 제시되지 않은 답변은 민원 응대에 바로 사용하지 않는다.
- 민원인의 주민등록번호, 연락처, 주소 등은 입력 시 마스킹한다.
- 법적 판단, 처분 가능성, 권리 제한 안내는 담당자 확인 후 사용한다.

### 4.4 뉴스 기사 수집

1. 관리자가 수집 키워드와 제외 키워드를 등록한다.
2. 매일 아침 예약 작업을 실행한다.
3. 사용자는 뉴스 화면에서 수집 결과를 확인한다.
4. 필요한 기사는 북마크하거나 팀에 공유한다.

운영 기준:

- 기사 전문 저장을 피하고 제목, 원문 링크, 요약 중심으로 제공한다.
- 수집 실패 시 관리자에게 알림을 보낸다.
- 키워드는 너무 넓게 설정하지 않고 업무 분야별로 나눈다.

## 5. 자주 발생하는 오류와 대응

### 5.1 프론트엔드 실행 오류

#### `npm run dev` 스크립트가 없음

원인:

- `package.json`에 Vite 실행 스크립트가 정의되지 않음

대응:

```json
{
  "scripts": {
    "dev": "vite"
  }
}
```

#### 포트 5173 사용 중

원인:

- 다른 Vite 개발 서버가 이미 실행 중

대응:

```powershell
npm run dev -- --port 5174
```

### 5.2 백엔드 실행 오류

#### `uvicorn` 명령을 찾을 수 없음

원인:

- 가상환경이 활성화되지 않았거나 패키지가 설치되지 않음

대응:

```powershell
cd backend
.\.venv\Scripts\activate
pip install fastapi uvicorn
```

#### `ModuleNotFoundError: app`

원인:

- 백엔드 엔트리 구조가 없거나 실행 위치가 잘못됨

대응:

- `backend/app/main.py`가 존재하는지 확인한다.
- `backend` 디렉터리에서 `uvicorn app.main:app --reload`를 실행한다.

### 5.3 엑셀 처리 오류

#### 컬럼을 찾을 수 없음

원인:

- 헤더 행이 다르거나 병합 셀이 포함됨
- 파일마다 컬럼명이 다름

대응:

- 업로드 전 첫 번째 행을 헤더로 정리한다.
- 컬럼 매핑 기능으로 유사 컬럼을 연결한다.
- 오류 리포트의 누락 컬럼 목록을 확인한다.

#### 대용량 파일 처리 실패

원인:

- API 요청 시간 초과 또는 메모리 부족

대응:

- 비동기 작업 큐로 처리한다.
- 파일 크기와 행 수 제한을 둔다.
- 처리 중간 상태를 작업 ID로 조회할 수 있게 한다.

### 5.4 챗봇 답변 오류

#### 근거 문서가 제시되지 않음

원인:

- 매뉴얼 인덱싱 실패
- 질문과 관련된 매뉴얼 내용 부족

대응:

- 매뉴얼 업로드 상태를 확인한다.
- 문서 파싱 로그를 확인한다.
- 키워드와 질문 표현을 구체화한다.
- 근거 없는 답변은 사용하지 않는다.

#### 개인정보가 답변에 포함됨

원인:

- 사용자가 민원 원문을 그대로 입력
- 마스킹 필터 미적용

대응:

- 입력 단계에서 개인정보 탐지 및 마스킹을 적용한다.
- 응답 로그에는 원문 전체를 저장하지 않는다.
- 운영자는 개인정보 포함 로그를 즉시 삭제하고 접근 이력을 확인한다.

### 5.5 뉴스 수집 오류

#### 예약 수집이 실행되지 않음

원인:

- 스케줄러 프로세스 미실행
- 시간대 설정 오류
- Redis 또는 작업 큐 장애

대응:

- 스케줄러 프로세스 상태를 확인한다.
- 서버 시간대와 `NEWS_COLLECT_TIME`을 확인한다.
- 작업 큐 연결 상태를 확인한다.

#### 중복 기사가 많음

원인:

- 제목 유사도 비교 미적용
- 동일 보도자료 기반 기사가 다수 수집됨

대응:

- URL, 제목 정규화, 발행일 기준 중복 제거를 적용한다.
- 제목 유사도 임계값을 조정한다.

## 6. 운영 체크리스트

일일 점검:

- 백엔드 API 헬스체크 정상 여부
- 뉴스 수집 작업 성공 여부
- 실패한 엑셀 작업 존재 여부
- 챗봇 매뉴얼 인덱스 상태

주간 점검:

- 업로드 파일 보관 기간 초과 데이터 삭제
- 감사 로그 이상 패턴 확인
- 뉴스 키워드 품질 확인
- 챗봇 답변 샘플 검토

월간 점검:

- DB 백업 복구 테스트
- 권한 계정 정리
- 민원 매뉴얼 최신 버전 반영
- 보안 패치와 의존성 업데이트 검토

## 7. 백업과 보관 정책

| 데이터 | 보관 기준 |
| --- | --- |
| 일정 데이터 | 업무 기준에 따라 장기 보관 |
| 일정 변경 이력 | 감사 목적에 따라 보관 |
| 엑셀 원본 파일 | 처리 완료 후 단기 보관 뒤 삭제 |
| 엑셀 결과 파일 | 다운로드 가능 기간 후 삭제 |
| 민원 대응 로그 | 최소 보관, 개인정보 마스킹 |
| 매뉴얼 파일 | 버전별 보관 |
| 뉴스 메타데이터 | 제목, 링크, 요약 중심 보관 |
| 감사 로그 | 보안 정책에 따라 별도 보관 |

## 8. 장애 대응 절차

1. 장애 범위를 확인한다.
2. 최근 배포, 설정 변경, 작업 실패 로그를 확인한다.
3. 사용자 영향 기능을 공지한다.
4. 임시 조치가 가능하면 우선 적용한다.
5. 원인을 분석하고 재발 방지 항목을 기록한다.
6. 운영 문서를 업데이트한다.

## 9. 배포 전 확인사항

- 환경 변수가 모두 설정되었는지 확인
- DB 마이그레이션 적용 여부 확인
- 관리자 계정과 기본 권한 설정 확인
- 파일 업로드 제한과 삭제 정책 확인
- LLM API 설정 및 뉴스 수집 설정 확인
- 헬스체크 엔드포인트 정상 여부 확인
- 프론트엔드 빌드 성공 여부 확인
- 주요 기능 수동 테스트 완료 여부 확인
